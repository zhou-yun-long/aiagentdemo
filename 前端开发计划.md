# 前端开发计划

> 本计划基于 `Treeify_开发需求文档.md`、`项目图片分析学习文档.md`、根目录 6 张项目图片以及当前 `frontend/` 实现整理。目标是把现有静态演示界面推进为可接入后端、可生成、可编辑、可执行、可导出的 Treeify 前端工作台。

## 1. 计划依据

### 1.1 需求文档给出的硬约束

| 来源 | 对前端的要求 |
| --- | --- |
| 技术架构 | React + TypeScript + Vite，组件化 SPA，支持 SSE 流式渲染 |
| 核心模块 | 项目管理、摘要 Agent、三阶段生成、RAG 知识库、用例管理、MCP 集成 |
| GeneratePanel | 自动/逐步模式、需求输入、PRD 上传、E1/E2/E3 进度、流式展示、Critic 评分、用例预览编辑 |
| 用例管理 API | 用例列表、创建、更新、删除、批量确认、导出 JSON/CSV/Excel |
| 摘要/RAG | 摘要状态展示、摘要历史、知识文档上传、知识库检索调试 |
| 错误码 | 前端需要处理参数错误、鉴权失败、项目不存在、摘要更新中、LLM 失败、任务超时等状态 |

### 1.2 图片与学习文档给出的产品形态

图片表达的前端不是普通表单系统，而是“高密度测试用例工作台”：

| 图片学习结论 | 前端落地方式 |
| --- | --- |
| AI 驱动用例生成 | 右侧 AI 助手 + GeneratePanel 流式生成 |
| 思维导图式用例管理 | 中央画布作为主工作区，节点呈现用例结构 |
| 全流程管理 | 生成、编辑、执行、评审、版本、导出在一个工作台闭环 |
| 执行状态可视化 | 顶部通过率、已测数量、节点状态、执行记录 |
| 版本备份与安全追溯 | 快照、历史版本、差异对比、回滚入口 |
| 多端/多类型标签 | 平台标签、测试类型、优先级、AI 来源标签 |

### 1.3 当前前端现状

| 文件 | 当前已有 | 主要缺口 |
| --- | --- | --- |
| `frontend/src/App.tsx` | 静态三栏工作台、主题切换、节点选中 | 无路由、无真实状态管理、无后端数据 |
| `frontend/src/components/Toolbar.tsx` | 顶部统计、工具栏、标签栏、导出/分享按钮 | 按钮多数未绑定行为，统计为静态值 |
| `frontend/src/components/OutlinePanel.tsx` | 大纲列表与选中联动 | 无折叠、搜索、过滤、懒加载 |
| `frontend/src/components/MindMapCanvas.tsx` | 固定坐标画布、曲线连接、节点选择 | 无拖拽、自动布局、缩放状态、节点 CRUD |
| `frontend/src/components/NodeCard.tsx` | 优先级、状态、AI 标签展示 | 无内联编辑、右键菜单、状态切换 |
| `frontend/src/components/AiAssistantPanel.tsx` | 静态输入框、静态解析表格 | 未接生成任务、SSE、引用当前节点、回填画布 |
| `frontend/src/data/mindMap.ts` | 演示节点数据与基础类型 | 缺项目、用例、生成任务、执行记录、快照等业务模型 |

## 2. 总体目标

前端首个可交付版本应完成一条核心闭环：

```text
选择/创建项目
  -> 输入需求或上传 PRD
  -> 启动 AI 三阶段生成
  -> 流式查看 E1/E2/E3 结果
  -> 确认或修改生成结果
  -> 写入思维导图用例树
  -> 编辑节点和执行状态
  -> 导出或保存版本快照
```

不优先做营销落地页。第一屏保持工作台形态，突出 Treeify 的核心差异：AI 生成结果能落到可管理、可执行、可追溯的测试用例资产中。

## 3. 开发原则

1. 保留当前图片学习到的高密度工作台，不做大面积宣传 Hero。
2. 前端视图模型与后端业务模型分层，后端 `test_cases` 不直接绑死画布布局。
3. 先完成单项目核心链路，再扩展多项目、知识库、MCP。
4. AI 助手服务当前项目、当前用例集或当前节点，不做孤立聊天窗口。
5. 所有异步任务都要有状态、错误、重试和取消入口。
6. 明暗主题从一开始保持可用，避免后期补主题造成颜色体系返工。

## 4. 技术基线调整

需求文档写的是 React 18、Vite 5、Ant Design、Zustand、React Query；当前 `frontend/package.json` 实际是 React 19、Vite 7，且未安装 Ant Design、Zustand、React Query。

建议先做一次技术基线确认：

| 项目 | 建议 |
| --- | --- |
| React / Vite | 保持当前 React 19 + Vite 7，除非后端或组件库存在兼容性问题 |
| UI 组件库 | 工作台核心控件继续自研 CSS，表单、弹窗、上传、表格可引入 Ant Design |
| 状态管理 | 引入 Zustand 管理本地工作台状态、生成状态、画布状态 |
| 服务端状态 | 引入 React Query 管理项目、用例、摘要、知识库接口缓存 |
| SSE | 封装 `useGenerateStream`，统一处理连接、事件分发、重连、关闭 |
| 测试 | 引入 Vitest + React Testing Library；关键生成流和数据转换需要单测 |

frontend/src/

  shared/

## 6. 核心数据模型

### 6.1 前端视图模型

继续保留 `MindNode` 作为思维导图专用模型，并补充以下字段：

| 字段 | 用途 |
| --- | --- |
| `caseId` | 关联后端测试用例 |
| `projectId` | 关联项目 |
| `collapsed` | 大纲和画布折叠 |
| `executionStatus` | not_run / running / passed / failed / blocked / skipped |
| `source` | ai / manual / imported |
| `version` | 用于快照和冲突检测 |
| `layout` | 存储 x/y、宽高、拖拽位置 |

### 6.2 后端接口模型

前端需要定义并维护以下类型：

| 类型 | 来源 |
| --- | --- |
| `Project` | `/api/v1/projects` |
| `ProjectSummary` | `/api/v1/projects/{id}/summary` |
| `GenerateTask` | `/api/v1/projects/{id}/generate` |
| `StageResult` | SSE `stage_done` |
| `GeneratedCase` | SSE `generation_complete` |
| `TestCase` | `/api/v1/projects/{id}/cases` |
| `KnowledgeDocument` | `/api/v1/projects/{id}/knowledge` |
| `ApiError` | 统一错误码规范 |

### 6.3 转换层

必须提供显式转换函数：

| 函数 | 职责 |
| --- | --- |
| `caseToMindNodes(testCase)` | 后端用例转思维导图节点 |
| `mindNodesToCase(nodes)` | 思维导图节点转后端用例 |
| `generatedCasesToMindNodes(cases)` | AI 生成结果批量写入画布 |
| `stageResultToPreview(result)` | E1/E2/E3 结果转预览视图 |

## 7. Sprint 计划

### Sprint 0：技术基线与设计对齐

周期：2-3 天

目标：把当前静态 Demo 稳定为可继续开发的前端工程底座。

| 任务 | 交付物 | 验收标准 |
| --- | --- | --- |
| 确认品牌名 | Treeify/KCase 命名决策记录 | 页面标题、Logo 文案、导出文件名前缀一致 |
| 依赖决策 | Zustand、React Query、Ant Design、测试库引入方案 | `package.json` 与技术文档一致 |
| 建立类型层 | `shared/types` | 项目、用例、生成任务、错误码类型可复用 |
| 建立 API 层 | `shared/api` | 能统一处理 baseUrl、错误码、JSON 解析 |
| 建立主题变量 | `theme.css` 或继续收敛 `app.css` | 亮色/暗色变量可扩展，截图中的主色与状态色稳定 |

### Sprint 1：工作台静态能力产品化

周期：1 周

目标：让现有图片工作台从静态展示变成可操作的本地工作台。

| 任务 | 范围 | 验收标准 |
| --- | --- | --- |
| 节点选择与编辑 | 节点标题、优先级、标签、状态 | 选中节点后可编辑并实时更新画布和大纲 |
| 节点 CRUD | 新增同级/下级、删除、移动 | 顶部工具栏按钮有真实行为 |
| 大纲增强 | 折叠、搜索、按状态过滤 | 大量节点时可快速定位 |
| 画布基础控制 | 缩放、居中、适配屏幕、隐藏大纲 | 缩放值真实生效，布局不跳动 |
| 主题完善 | 明暗主题 | 白色/黑色主题下节点、连线、标签、表格均可读 |

### Sprint 2：项目与用例接口接入

周期：1-2 周

目标：完成项目和用例的基本持久化。

| 任务 | 接口 | 验收标准 |
| --- | --- | --- |
| 项目列表/创建 | `GET/POST /api/v1/projects` | 可创建项目并进入工作台 |
| 项目详情 | `GET /api/v1/projects/{id}` | 页面刷新后能恢复当前项目 |
| 用例列表 | `GET /api/v1/projects/{id}/cases` | 后端用例可渲染为思维导图 |
| 用例保存 | `POST/PUT /api/v1/cases` | 编辑节点后可保存到后端 |
| 用例删除 | `DELETE /api/v1/cases/{caseId}` | 删除后画布、大纲、统计同步更新 |
| 错误处理 | 统一错误码 | 401、403、404、409、422、408 有明确提示 |

### Sprint 3：AI 三阶段生成与 SSE

周期：1-2 周

目标：完成需求输入到 AI 生成结果预览的主链路。

| 任务 | 接口/事件 | 验收标准 |
| --- | --- | --- |
| GeneratePanel | 本地组件 | 支持自动/逐步模式、需求输入、PRD 上传入口 |
| 创建生成任务 | `POST /api/v1/projects/{id}/generate` | 返回 taskId 后进入流式状态 |
| SSE Hook | `GET /api/v1/generate/{taskId}/stream` | 能接收 e1_start、e1_chunk、stage_done、generation_complete、error |
| 阶段展示 | E1/E2/E3 | 流式内容实时追加，阶段完成后结构化展示 |
| 逐步确认 | `POST /api/v1/generate/{taskId}/confirm` | E1/E2 可确认继续，支持修改意见 |
| Critic 展示 | `critic_score` | 分数低于 70 显示质量警告和重试结果 |
| 取消任务 | `POST /api/v1/generate/{taskId}/cancel` | 用户可停止生成并关闭 SSE |

### Sprint 4：生成结果回填与 AI 助手

周期：1 周

目标：把 AI 结果真正落入思维导图和用例管理。

| 任务 | 范围 | 验收标准 |
| --- | --- | --- |
| CasePreview | 生成用例列表 | 支持内联编辑标题、前置条件、步骤、预期、优先级 |
| 批量确认 | `POST /api/v1/cases/batch-confirm` | 确认后写入后端并渲染到画布 |
| AI 助手上下文 | 当前项目/节点/选区 | AI 续写能引用当前节点内容 |
| AI 解析表格 | 自然语言 -> 表格 | 解析结果可选择性回填到画布 |
| 重新生成本阶段 | 当前阶段 + 修改意见 | 选中节点可触发局部重新生成 |

### Sprint 5：执行、评审、版本与导出

周期：1-2 周

目标：补齐宣传图中的“执行与评审、版本备份、安全追溯”。

| 任务 | 范围 | 验收标准 |
| --- | --- | --- |
| 执行状态 | not_run/running/passed/failed/blocked/skipped | 顶部通过率和已测数量实时计算 |
| 执行记录 | 节点或用例维度 | 可查看最近执行结果、失败原因、执行人、时间 |
| 快照当前结果 | 工作台快照 | 可保存快照并恢复 |
| 版本历史 | 用例版本/项目摘要版本 | 可查看历史版本与当前版本差异 |
| 导出用例 | JSON/CSV/Excel/Markdown | 导出内容包含标题、前置条件、步骤、预期、优先级、标签、执行状态 |
| 分享只读视图 | 用例分享 | 分享链接可只读查看，不允许编辑 |

### Sprint 6：摘要、知识库与集成设置

周期：1-2 周

目标：实现 P1/P2 能力，支撑更完整的项目质量管理。

| 任务 | 接口 | 验收标准 |
| --- | --- | --- |
| 摘要状态展示 | `GET /api/v1/projects/{id}/summary` | 工作台可显示当前项目摘要和版本 |
| 摘要历史/回滚 | history / rollback | 可查看历史摘要并回滚 |
| PRD 上传 | `POST /api/v1/projects/{id}/documents` | 上传后显示摘要更新状态 |
| 知识库管理 | `/knowledge` | 可上传、查看、删除知识片段 |
| 语义检索调试 | `/knowledge/search` | 输入查询能看到 RAG Top-N 结果 |
| MCP 设置页 | MCP 集成接口 | 可配置 Jira/禅道/CI 等外部工具入口 |

## 10. 验收清单

### P0 MVP 验收

1. 用户可以创建项目并进入工作台。
2. 用户可以输入需求并启动 AI 生成。
3. 前端可以实时展示 E1/E2/E3 流式结果。
4. 逐步模式下用户可以确认阶段继续。
5. 生成用例可以编辑并批量确认。
6. 确认后的用例可以在思维导图中展示。
7. 用户可以编辑、删除、移动用例节点。
8. 通过率、已测数量、节点状态可以根据数据实时更新。
9. 页面刷新后能从后端恢复项目和用例。
10. 常见错误码有明确提示，不出现静默失败。

### P1 验收

1. 支持项目摘要展示、历史查看和回滚。
2. 支持上传知识文档并查看入库状态。
3. AI 生成时能体现项目摘要与历史用例上下文。
4. 支持导出用例。
5. 支持版本快照和基础差异查看。

### P2 验收

1. 支持 MCP/Jira/禅道/CI 集成设置。
2. 支持分享只读视图。
3. 支持更完善的自动布局、拖拽布局和大量节点性能优化。

## 11. 风险与决策点

| 风险/决策点 | 当前判断 | 处理建议 |
| --- | --- | --- |
| 品牌名不一致 | 宣传图为 KCase，需求文档为 Treeify | 开发前先确认最终产品名 |
| 依赖版本不一致 | 需求文档与实际 `package.json` 不一致 | Sprint 0 决策并更新文档或依赖 |
| 后端接口未完全就绪 | 前端真实接入可能被阻塞 | 先用 MSW 或本地 mock 固定接口契约 |
| 思维导图固定布局 | 数据变多后会重叠 | P0 可用固定布局，P2 引入自动布局/拖拽布局 |
| SSE 事件格式变动 | 前后端联调风险较高 | 提前定义事件类型和样例数据，写单测覆盖解析 |
| AI 输出结构不稳定 | 影响 CasePreview 和回填 | 前端只接受校验后的结构化结果，异常进入人工编辑 |
| 深色主题对比度 | 图片中深色主题要求高 | 每个 Sprint 都保留明暗主题视觉检查 |

## 12. 最近一次可执行任务列表

按当前仓库状态，建议下一轮直接执行这些任务：

1. 新建 `shared/types`，把 `MindNode` 扩展为正式视图模型。
2. 新建 `shared/api`，封装 `request`、`ApiError`、错误码映射。
3. 引入 Zustand，建立 `workspaceStore` 管理节点、选中项、主题、面板状态。
4. 将 `mindNodes` 从静态常量迁移为 store 初始数据。
5. 给 `Toolbar` 的新增、删除、上移、下移、导出按钮绑定本地行为。
6. 给 `NodeCard` 和 `SelectionBar` 增加编辑弹窗或内联编辑。
7. 新建 `generationStore` 和 `useGenerateStream`，先用 mock SSE 数据跑通 E1/E2/E3 展示。
8. 新建 `CasePreviewTable`，把 mock 生成用例回填到画布。

完成以上任务后，前端就从“静态截图复刻”进入“可操作 MVP 骨架”，后续接真实后端会更顺。
