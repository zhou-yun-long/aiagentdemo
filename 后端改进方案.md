# 后端改进方案

> 本方案基于 `Treeify_开发需求文档.md`、`前端开发计划.md`、`项目图片分析学习文档.md` 和当前 Spring AI Demo 后端代码整理。目标是把现有通用 AI Agent Demo 改造成 speccase 可用的后端平台，并保证前后端 API、状态、数据模型、SSE 事件和错误码对齐。

## 1. 当前后端现状

当前后端已经具备一部分可复用的 AI 基础能力，但还不是 speccase 业务后端。

| 模块 | 当前已有 | 与 speccase 的差距 |
| --- | --- | --- |
| `ChatController` | `/api/chat` 普通对话和流式对话 | 面向通用聊天，不是项目级生成任务 |
| `ManagerController` | 清空记忆、切换模型、MCP 连接管理 | 缺项目、用例、摘要、知识库业务管理 |
| `AgentCore` | 意图识别、RAG 注入、ChatClient 调用、工具调用 | 缺 E1/E2/E3/Critic 专用编排与任务状态 |
| `SubAgentManager` | 动态创建 SubAgent | 缺 speccase 固定角色 Agent 和结构化输出校验 |
| `RagService` | classpath 知识库、向量检索、长上下文降级 | 缺项目级知识库、历史用例入库、pgvector 持久化 |
| MCP | MCP Client/Server 基础能力 | 缺 Jira/禅道/CI 等业务集成配置模型 |
| DTO | `ChatResponse` 使用 success/reply/error | 与需求文档 `{ code, message, data }` 不一致 |
| 持久化 | 当前无业务库访问层 | 缺项目、用例、生成任务、执行记录、版本快照等表 |

结论：后端改进不能只在 `AgentCore` 上补接口，而应新增 speccase 业务层，把通用 AI 能力沉到可复用基础设施层。

## 2. 前后端对齐目标

后端第一阶段必须支撑前端 P0 MVP：

```text
项目创建/选择
  -> 加载项目用例
  -> 创建 AI 生成任务
  -> SSE 推送 E1/E2/E3/Critic
  -> 前端预览并批量确认用例
  -> 用例保存、编辑、删除、执行状态更新
  -> 导出或保存快照
```

需要前后端优先对齐的契约：

| 契约 | 后端责任 | 前端依赖 |
| --- | --- | --- |
| 统一响应 | 所有 REST API 返回 `{ code, message, data, requestId }` | `shared/api` 统一解析错误 |
| 错误码 | 按需求文档错误码扩展 | UI 能映射为明确提示 |
| 项目模型 | `Project` 字段稳定 | 项目列表、工作台标题 |
| 用例模型 | `TestCase` 字段稳定 | `caseToMindNodes` / `mindNodesToCase` |
| 生成任务模型 | `GenerateTask` 字段稳定 | `generationStore` 状态 |
| SSE 事件 | 事件名、payload、结束条件稳定 | `useGenerateStream` 可可靠解析 |
| 状态枚举 | 任务状态、执行状态、优先级一致 | 顶部统计、节点状态、筛选 |
| 分页过滤 | 列表接口参数一致 | React Query 缓存 key 稳定 |

## 4. 统一响应与错误码

### 4.1 统一响应格式

后端应新增统一响应对象，替代业务 API 里的 `ChatResponse`：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "requestId": "req-20260425-abc123"
}
```

分页统一为：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 135
}
```

`/api/chat` 可以暂时保留旧格式，speccase 新接口统一走 `/api/v1/**`。

### 4.2 错误码对齐

| code | HTTP | 前端处理 | 后端触发场景 |
| --- | --- | --- | --- |
| 0 | 200/201/202 | 成功 | 正常返回 |
| 1001 | 400 | 表单字段提示 | 参数缺失、格式错误、JSON 解析失败 |
| 1002 | 401 | 跳转登录 | 未登录或 token 过期 |
| 1003 | 403 | 权限不足提示 | 非项目成员访问 |
| 2001 | 404 | 返回项目列表或显示空态 | 项目、用例、任务不存在 |
| 2002 | 409 | 显示“摘要更新中”并轮询 | 摘要任务未完成 |
| 3001 | 422 | 可重试提示 | LLM 调用失败、结构化输出解析失败 |
| 3002 | 422 | 质量警告 | Critic 重试次数超限 |
| 3003 | 408 | 超时提示 | 生成任务超时 |
| 4001 | 500 | 非阻塞提示 | 向量化入库失败（P1），可异步重试 |

后端需要 `GlobalExceptionHandler` 把业务异常、校验异常、未知异常统一转换为上述格式。

## 7. SSE 事件契约

前端计划中的 `useGenerateStream` 依赖稳定事件名和 JSON payload。后端不要直接推裸字符串，统一推 `ServerSentEvent<GenerateSseEvent>`。

### 7.1 通用事件格式

```json
{
  "event": "stage_chunk",
  "taskId": "a3f2c1d0-0000-4000-8000-abcdef123456",
  "stage": "e1",
  "sequence": 12,
  "timestamp": "2026-04-25T10:00:00+08:00",
  "payload": {}
}
```

### 7.2 事件列表

| event | stage | payload | 前端动作 |
| --- | --- | --- | --- |
| `stage_started` | e1/e2/e3/critic | `{ stage }` | 点亮阶段进度 |
| `stage_chunk` | e1/e2/e3 | `{ content }` | 追加流式文本 |
| `stage_done` | e1/e2/e3/critic | `{ result, needConfirm }` | 结构化展示阶段结果 |
| `generation_complete` | null | `{ criticScore, cases }` | 打开 CasePreview |

`stage_done.payload.needConfirm=true` 表示逐步模式需要前端显示确认按钮；`critic` 阶段也使用 `stage_started` 和 `stage_done`，不再单独定义 `critic_done`。

## 8. 三阶段生成编排方案

### 8.1 OrchestrationService 职责

`OrchestrationService` 是 speccase 后端核心，不直接复用聊天接口：

1. 创建并持久化 `generation_tasks`。
2. 读取项目摘要、历史用例上下文、用户输入、引用用例。P0 历史上下文来自 MySQL/关键词过滤，P1 替换为 RAG Top-N。
3. 调用 E1 需求分析 Agent，产出结构化 JSON。
4. 在逐步模式下进入 `wait_confirm`。
5. 调用 E2 测试对象 Agent。
6. 调用 E3 用例生成 Agent。
7. 调用 Critic Agent 评分和必要重试。
8. 推送 SSE 并记录 `generation_events`。
9. 完成后返回 `GeneratedCase[]`，等待前端批量确认。

### 8.2 固定 Agent 角色

| Agent | 输入 | 输出 | 校验 |
| --- | --- | --- | --- |
| E1 需求分析 | 项目摘要 + 历史用例上下文 + 用户输入 | `E1Result` JSON | P0 使用 MySQL/关键词上下文，P1 使用 RAG；必须包含目标、动作、系统行为、约束 |
| E2 测试对象 | `E1Result` | `TestObject[]` | 每项必须有 name/type/dimensions/priority |
| E3 用例生成 | `TestObject[]` + 约束 | `GeneratedCase[]` | title/precondition/steps/expected/priority 必填 |
| Critic | E2 + E3 | `CriticReport` | score 0-100，issues 数组 |

### 8.3 结构化输出防线

AI 输出不能直接入库，应经过三层校验：

1. JSON 解析：失败触发一次“修复 JSON”提示。
2. Schema 校验：字段缺失、类型错误、空步骤直接失败或重试。
3. 业务校验：priority 枚举、steps 非空、expected 不允许过度模糊。

只有通过校验的 `GeneratedCase[]` 才推给前端 `CasePreview`。

## 9. RAG 与摘要改进

当前 `RagService` 读取 classpath 静态 txt，后续需要演进为项目级 RAG。

### 9.1 P0 过渡方案

P0 可先不接 pgvector：

| 能力 | 实现 |
| --- | --- |
| 项目摘要 | 先从 `project_summaries` 当前版本读取，没有则为空 |
| 历史用例上下文 | MySQL 中按项目取最近确认用例 Top-20，简单关键词过滤 |
| 知识库 | 保留现有 `RagService` 作为全局兜底 |

### 9.2 P1 正式方案

| 能力 | 实现 |
| --- | --- |
| 文档入库 | 上传 PRD/规范后切片、向量化、写入 `knowledge_chunks` |
| 历史用例入库 | 批量确认用例后异步向量化 |
| 混合检索 | project_id 过滤 + pgvector + 全文检索 |
| 摘要续写 | 新文档上传后调用 SummaryAgent 更新版本 |
| 摘要 Critic | 校验旧模块不丢失、新功能已加入 |

## 10. 执行、版本、导出对齐

### 10.1 执行状态

前端计划使用：

```text
not_run / running / passed / failed / blocked / skipped
```

后端数据库、DTO、过滤参数必须使用相同枚举。顶部统计计算规则：

| 指标 | 计算 |
| --- | --- |
| total | 当前项目用例数 |
| measured | 非 `not_run` 用例数 |
| passed | `passed` 用例数 |
| passRate | `passed / measured`，measured 为 0 时返回 0 |

### 10.2 快照与版本

| API | 说明 |
| --- | --- |
| `POST /api/v1/projects/{projectId}/snapshots` | 保存当前用例树快照 |
| `GET /api/v1/projects/{projectId}/snapshots` | 快照列表 |
| `GET /api/v1/snapshots/{snapshotId}` | 快照详情 |
| `POST /api/v1/snapshots/{snapshotId}/restore` | 恢复快照 |

P0 可先只做 `test_cases.version` 乐观锁；P1 再补完整快照。

### 10.3 导出

`POST /api/v1/cases/export` 请求：

```json
{
  "projectId": 1,
  "format": "markdown",
  "caseIds": [1001, 1002],
  "includeExecution": true
}
```

返回小文件可直接给下载 URL；大文件建议走 `export_tasks` 异步任务。

## 11. 安全与配置

当前 `application.properties` 使用环境变量读取 API Key，这是正确方向。后端改进时需要补齐：

| 项目 | 建议 |
| --- | --- |
| 密钥 | 严禁把 `.env` 或真实 API Key 入库 |
| CORS | Vite dev server 跨域白名单，可配置 `app.cors.allowed-origins` |
| 鉴权 | P0 可用 mock user，P1 接 JWT/session |
| 文件上传 | 限制大小 10MB、类型 `.pdf/.docx/.md/.txt` |
| 请求日志 | 输出 requestId、userId、projectId、taskId |
| 任务取消 | SSE 断开不等于任务取消，必须显式 cancel |
| 并发限制 | 单用户/单项目同时生成任务数量限制 |

## 13. 联调验收标准

### P0 联调必须通过

1. 前端调用 `GET /api/v1/projects` 能拿到统一响应。
2. 前端创建项目后能进入工作台并刷新恢复。
3. 前端加载用例后能生成 `MindNode` 树。
4. 前端编辑节点保存后，后端返回更新后的 `version`。
5. 前端创建生成任务后能拿到 `taskId` 和 `streamUrl`。
6. SSE 能稳定推送 `stage_started`、`stage_chunk`、`stage_done`、`generation_complete`。
7. 逐步模式下 E1/E2 进入 `wait_confirm`，确认后继续。
8. `generation_complete.payload.cases` 可直接进入 `CasePreviewTable`。
9. 批量确认后，前端重新拉取用例能看到新增用例。
10. LLM 失败、任务超时、用户取消都能产生明确错误事件和 REST 状态。
