## 一、Treeify 项目深度追问

### 架构设计

#### 🔴 为什么把用例生成拆成三个阶段？不直接一步生成？

**答题方向：**

一步生成属于黑盒产出，问题难以定位：分不清是需求理解错误、对象拆解遗漏还是场景覆盖不全。

三阶段拆分后每一步都有独立可校验产出，可干预、可修正：

E1 校验需求理解、E2 补充测试对象、E3 生成最终用例，避免最终大批量返工。

核心价值：**可解释性、可人工干预、分层 Debug、质量可控**。

------

#### 🔴 Sub-agent 和多智能体的区别？为什么不用多智能体？

**答题方向：**

- 多智能体：多个 Agent 平等并行协作，适合任务独立、可并行的场景
- Sub-agent：主 Agent 统一调度，按需串行调用专属子节点，强流程串行

测试用例生成是严格 **E1→E2→E3 强串行依赖**，上一步输出是下一步输入，并行无收益还会大幅提升调度复杂度。

Sub-agent 天然契合 SpringAI 原生能力，无需引入 LangGraph、AutoGen 等重型框架，轻量化、易维护、稳定性更高。

------

#### 🔴 摘要 Agent 的续写模式如何防止历史模块信息丢失？

**答题方向：**

三层兜底保障：

1. **结构化强约束**：系统 Prompt 强制固定输出结构（核心业务、模块列表、模块依赖、待测重点），模块化条目输出，避免合并丢失
2. **Critic 校验机制**：更新摘要后自动对比新旧模块清单，缺失则触发重新生成
3. **历史版本存档**：每次摘要更新持久化历史记录，信息丢失可一键回滚恢复

------

#### 🔴 Critic 评审不通过怎么重试？如何防止死循环？

**答题方向：**

不做无意义盲重试，**携带评审缺陷报告**（对象缺失、路径不全、用例模糊）作为额外上下文二次生成，精准修正问题。

防死循环策略：

固定最大重试次数（上限 2 次），超出阈值直接返回当前结果，并标记质量警告，不阻塞主任务流程。

------

#### 🟡 逐步模式下后端怎么挂起等待用户确认？

**答题方向：**

基于**任务状态机**设计：

E1 执行完成后，修改任务状态为 `wait_confirm_e1`，编排器暂停后续流程，实现任务挂起。

前端点击确认后，调用确认接口流转状态，后端继续执行 E2/E3。

超时容错：10 分钟未操作自动标记任务取消，释放服务资源。

------

#### 🟡 为什么新 PRD 不入 RAG 而是直接全文给 LLM？

**答题方向：**

RAG 适合**跨文档、海量知识库、长期沉淀**的知识检索，输出碎片化片段；

单份独立 PRD 需要完整上下文连贯理解，片段召回容易断章取义。

当前主流大模型上下文窗口充足，单文档全文投喂不会超限，理解准确度更高。

总结：**单文档一次性分析直推 Prompt，多文档长期复用才接入 RAG**。

------

#### 🟡 pgvector 混合检索的权重 0.7/0.3 是怎么定的？

**答题方向：**

属于业务场景经验最优值。

- 向量检索（0.7）：擅长语义理解、同义描述、业务需求模糊匹配
- BM25 关键词检索（0.3）：精准匹配接口名、专业术语、枚举、固定字段

测试用例场景语义匹配为主、精确关键词为辅，因此向量权重更高。

支持后续 A/B 测试、人工评估召回效果，动态调优权重配比。

------

#### 🟡 Critic 评审的四个维度是怎么设计的？权重怎么分配？

**答题方向：**

表格







|      维度      | 权重 |            设计理由            |
| :------------: | :--: | :----------------------------: |
| 测试对象覆盖率 | 40%  | 漏测为最高风险问题，优先级最高 |
| 路径类型完整性 | 30%  |  缺异常 / 边界场景是普遍短板   |
|  用例可执行性  | 20%  |   步骤、预期模糊导致无法落地   |
|   重复冗余率   | 10%  |   影响效率，但不影响核心覆盖   |

综合得分＜70 自动触发重试，阈值为工程经验值，可按需灵活调整。

------

#### 🟢 双库架构（MySQL + pgvector）怎么设计的？为什么不用一个库？

**答题方向：**

- MySQL：存储核心业务数据（项目、任务、用例、用户），CRUD 稳定、事务完善、运维成熟
- pgvector（PostgreSQL 扩展）：专注向量存储、语义检索、混合查询

两库通过 `project_id` 业务字段关联，应用层组装数据，不做跨库联表查询。

拆分原因：单一数据库无法同时兼顾**事务业务能力 + 高性能向量检索**，拆分架构各司其职，性能与扩展性更强。

------

#### 🟢 SSE 和 WebSocket 有什么区别？为什么选 SSE？

**答题方向：**

表格







|  对比项  |          SSE          |      WebSocket       |
| :------: | :-------------------: | :------------------: |
| 通信方向 |    服务端单向推送     |    全双工双向通信    |
| 底层协议 |       基于 HTTP       |     独立 WS 协议     |
| 实现成本 |       轻量简单        | 复杂、需自主管理心跳 |
| 断连处理 |     原生自动重连      |  手动实现重连、保活  |
| 适用场景 | AI 流式输出、消息推送 |  实时聊天、协同编辑  |

AI 生成场景只需**后端单向流式输出**，SSE 完全满足需求，更轻量、更易开发部署。

## 二、LLM 基础原理

### Transformer 与注意力机制

#### 🔴 什么是 Attention 机制？为什么 Transformer 用它替代 RNN？

**答题方向：**

注意力机制允许模型在计算每个 Token 时，全局加权关注序列中所有上下文信息，而非局限于局部相邻内容。

核心公式：

\(Attention(Q, K, V) = softmax(\frac{QK^T}{\sqrt{d_k}}) \cdot V\)

RNN 核心缺陷：

1. 串行时序计算，无法并行，训练效率极低
2. 长序列梯度消失，远距离依赖建模困难

Transformer 优势：

1. 自注意力全局建模，长依赖友好
2. 全量并行计算，训练与推理效率大幅提升

------

#### 🟡 什么是 token？中文和英文的 token 效率有什么差异？

**答题方向：**

Token 是大模型文本切分的最小处理单元，基于 BPE 字节对编码实现。

- 英文：单词粒度切分，单单词大多 1 个 token
- 中文：单汉字多为 1 个 token，信息密度更高

同等语义信息量下，**中文更省 Token、调用成本更低**；不同模型分词器存在差异，实际消耗需实测。

------

#### 🟡 什么是上下文窗口（Context Window）？超出了怎么办？

**答题方向：**

上下文窗口是模型单次可处理的最大 Token 总量（输入 + 输出总和）。

主流规格：32K / 128K / 200K 上下文。

超限解决方案：

1. 截断策略：舍弃早期历史，保留最新对话 / 文档
2. 摘要压缩：通过摘要 Agent 浓缩长文本，精简上下文
3. RAG 解耦：长文档向量化入库，按需片段检索，不全量灌入
4. 滑动窗口：固定保留最近 N 轮对话，控制长度

------

#### 🟢 什么是 Temperature 和 Top-p？怎么选择？

**答题方向：**

- Temperature：控制生成随机性与发散度。值越低越确定、越高越创意
- Top-p（核采样）：按累计概率阈值筛选候选 Token，控制输出多样性

业务实践：

- 结构化 JSON、标准化输出：Temperature = 0 ~ 0.1

- 常规文案、问答生成：Temperature = 0.3 ~ 0.7

- 创意文案、发散写作：Temperature = 0.7 ~ 1.0

  

  常规场景二选一配置，避免同时叠加调节。

------

### 模型能力与限制

#### 🔴 什么是幻觉（Hallucination）？你在项目中怎么应对？

**答题方向：**

幻觉指大模型基于概率生成，输出看似合理、实则虚假、捏造的错误内容。

项目落地解决方案：

1. RAG 注入真实业务资料，基于事实生成
2. 强结构化输出约束，限制自由发挥
3. Critic 评审校验内容合理性，错误自动重试
4. 项目摘要固定注入，统一业务认知，减少错误假设

------

#### 🟡 什么是 Few-shot 和 Zero-shot？什么时候用？

**答题方向：**

- Zero-shot：仅任务描述，无示例，直接生成结果
- Few-shot：Prompt 中附带标准输入输出样例，引导模型对齐格式与规则

使用原则：

- 简单任务、强能力大模型：Zero-shot 够用
- 定制格式、复杂规则、非标输出：必须 Few-shot
- 样例控制在 2~5 个，避免 Token 冗余、收益递减

------

#### 🟢 Fine-tuning 和 RAG 的区别？什么时候用 Fine-tuning？

**答题方向：**

表格







|   对比   |          RAG           |       微调 Fine-tuning       |
| :------: | :--------------------: | :--------------------------: |
| 知识更新 |      实时动态更新      |      需重新训练，周期长      |
| 实现成本 |     低、轻量易落地     |     算力 / 数据 / 成本高     |
| 核心用途 | 实时知识检索、事实问答 | 固化风格、统一格式、领域适配 |
| 可解释性 |   高，可追溯召回来源   |        黑盒、不可追溯        |

适用场景：

优先 RAG 做知识增强；需要**固定输出风格、统一话术、行业专属范式**且标注数据充足时，再考虑微调。

## 三、Prompt 工程

#### 🔴 System Prompt 和 User Prompt 有什么区别？如何设计 System Prompt？

**答题方向：**

- System Prompt：全局角色定义、行为约束、输出规范，全局对话生效
- User Prompt：单轮用户具体业务请求，单次生效
- Assistant：历史模型回复，用于维持多轮上下文关联

系统提示词设计规范：

1. 明确角色定位与业务职责
2. 强制约束输出格式、禁止冗余内容
3. 写明禁止行为、边界限制
4. 注入固定背景信息、业务规则

------

#### 🔴 如何让 LLM 稳定输出 JSON 格式？有哪些技巧？

**答题方向：**

1. 系统提示词强约束：只输出合法 JSON，禁止 Markdown、注释、多余文案
2. 提供标准 JSON Schema 结构约束
3. 降低随机性：Temperature 置 0，保证确定性输出
4. 模型专属配置：开启 JSON 强制输出模式
5. Few-shot 样例引导，提供标准输入输出模板
6. 代码层兜底：正则提取 + 异常捕获 + 格式错误重试

------

#### 🟡 什么是 Chain of Thought（CoT）？什么时候用？

**答题方向：**

思维链，引导模型**先分步推理、再给出结论**，通过 step by step 拆解提升复杂问题准确率。

适用场景：

逻辑判断、规则校验、评审打分、多步骤分析类任务。

简单格式化输出、短内容生成无需 CoT，只会浪费 Token。

------

#### 🟡 Prompt 注入攻击是什么？怎么防范？

**答题方向：**

用户通过恶意特殊指令，覆盖、篡改、绕过系统预设约束，越权执行风险行为。

防范手段：

1. 输入内容过滤，拦截高危指令
2. 指令与用户内容强隔离，边界分隔
3. 输出结构化校验，拒绝异常内容
4. 禁止模型直接操作外部接口、命令执行

------

#### 🟢 什么是 Prompt 模板（Prompt Template）？如何管理？

**答题方向：**

带占位变量的标准化提示词模板，实现内容动态拼接、统一复用。

管理方案：

- 小型项目：代码常量、配置文件维护
- 中大型项目：数据库持久化，支持版本管理、A/B 测试、在线热更新

## 四、RAG 检索增强生成

#### 🔴 RAG 的完整流程是什么？

**答题方向：**

1. 离线阶段：文档解析 → 文本分块 Chunking → Embedding 向量化 → 存入向量库
2. 在线阶段：用户问题 → Query 向量化 → 向量 / 混合检索 TopK → 可选 Rerank 重排 → 拼接上下文进 Prompt → LLM 生成答案

------

#### 🔴 文档分块（Chunking）策略有哪些？各自优缺点？

**答题方向：**

表格







|     策略     |          优点          |           缺点           |         适用场景         |
| :----------: | :--------------------: | :----------------------: | :----------------------: |
| 固定长度分块 |   实现简单、均匀可控   | 切割语义、破坏段落完整性 |       通用兜底方案       |
| 标题层级分块 | 语义完整、业务边界清晰 |       块大小不可控       | Markdown/PRD 结构化文档  |
| 滑动重叠分块 |    减少边界信息丢失    |    存储冗余、成本上升    | 长文本、强上下文依赖场景 |
|   语义分块   |       分割最合理       |     计算耗时、成本高     |     高精度知识库场景     |

------

#### 🔴 什么是混合检索？为什么比纯向量检索更好？

**答题方向：**

混合检索 = **向量语义检索 + BM25 关键词检索** 结果融合排序。

纯向量短板：专业名词、接口名、枚举、关键字匹配弱；

纯关键词短板：无法理解语义、同义表达无法召回。

两者互补：语义模糊匹配 + 精准关键词锁定，大幅提升召回准确率。

------

#### 🟡 Embedding 模型如何选择？有哪些常用的？

**答题方向：**

表格







|        模型        |      特点      |       适用场景       |
| :----------------: | :------------: | :------------------: |
|    bge-large-zh    |  中文效果最优  |  中文业务知识库首选  |
|      m3e-base      |  轻量、低成本  |  中小项目、本地部署  |
| 官方通用 Embedding | 稳定、接口成熟 | 快速上线、云服务场景 |

选型原则：中文业务优先中文专属 Embedding，追求性价比用轻量模型，高精度选用大尺寸向量模型。

------

#### 🟡 什么是 Reranking？什么时候需要？

**答题方向：**

重排序，对向量粗排召回的 TopK 结果，使用高精度打分模型二次筛选、重新排序，过滤无关片段。

作用：提升上下文质量、减少无效内容灌入 LLM。

适用：文档量大、召回杂乱、问答精度要求高的场景。

------

#### 🟢 RAG 效果不好怎么排查？

**答题方向：**

三层排查法：

1. 召回层：检查 Chunk 质量、检索相似度、分块合理性
2. 内容层：文档脏数据、碎片化、关键信息被切割
3. 生成层：上下文拼接位置、Prompt 引导、模型是否引用召回内容

------

## 五、Agent 与工具调用

#### 🔴 什么是 AI Agent？和普通 LLM 调用有什么区别？

**答题方向：**

普通 LLM：单次单次输入输出，无自主规划、无外部能力。

AI Agent = 大模型 + 工具调用 + 任务规划 + 记忆管理 + 循环迭代。

可自主拆分任务、调用外部能力、根据返回结果动态调整执行步骤，复杂任务闭环完成。

------

#### 🔴 什么是 Function Calling / Tool Use？如何实现？

**答题方向：**

函数调用，模型识别意图后，按约定 JSON 格式输出工具调用参数，业务层执行工具并回传结果，模型结合返回内容生成最终答案。

SpringAI 可通过函数回调快速注册工具，原生标准化接入。

------

#### 🟡 ReAct 框架是什么？

**答题方向：**

Reasoning + Acting 推理行动框架，通过 思考→行动→观察 循环，让模型分步决策、调用工具、迭代执行。

优势：过程透明、易调试，是行业主流轻量 Agent 实现方案。

------

#### 🟡 什么是 MCP（Model Context Protocol）？和 Function Calling 的区别？

**答题方向：**

MCP 是统一上下文通信协议，标准化模型与外部工具、数据源的交互。

- Function Calling：厂商私有实现，封闭性强
- MCP：跨模型、跨系统通用标准，支持动态工具发现、解耦扩展

------

#### 🟢 Agent 规划失败了怎么处理？有哪些容错策略？

**答题方向：**

1. 限制最大重试次数，避免死循环
2. 失败降级，输出可用结果并标注风险
3. 任务超时熔断，防止资源占用
4. 关键节点人工确认介入
5. 状态持久化，支持断点续跑

------

## 六、Spring AI 框架

#### 🔴 Spring AI 的核心组件有哪些？

------

#### 🔴 Spring AI 如何实现流式输出？

**答题方向：**

通过 `chatClient.stream()` 返回 Flux 流式数据流，结合 SSE 协议，以长连接持续向前端推送增量内容，实现打字机效果。

#### 🟡 Spring AI 的 Advisor 是什么？QuestionAnswerAdvisor 怎么用？

Advisor 是 SpringAI 切面增强机制，在 LLM 调用前后统一处理通用能力。

`QuestionAnswerAdvisor` 是官方一键 RAG 增强，自动完成问题向量化、向量检索、上下文注入，快速实现知识库问答。

## 七、向量数据库

#### 🔴 向量相似度计算有哪几种方式？各自适用场景？

**答题方向：**

- 余弦相似度：只关注方向，文本语义检索**通用首选**
- 欧式距离：关注向量数值差异，适合数值型场景
- 点积 / 内积：适用于归一化向量，计算效率高

pgvector 分别提供对应操作符，适配不同业务检索需求。

------

#### 🟡 pgvector 的 IVFFlat 索引是什么？和 HNSW 有什么区别？

**答题方向：**

- IVFFlat：聚类索引，构建快、内存占用低，适合中小数据量
- HNSW：多层图索引，查询极速、内存开销高，适合高并发大数据量

项目数据量级不大，IVFFlat 完全够用，运维成本更低。

------

#### 🟢 向量数据库和传统数据库的本质区别是什么？

**答题方向：**

- 传统数据库：精准匹配、结构化事务、CRUD 业务

- 向量数据库：ANN 近似检索、语义匹配、相似度查询

  

  核心差异：

  精准查询 vs 语义模糊相似查询

  。

## 八、AI 应用工程实践

#### 🔴 AI 应用和传统应用在架构上有哪些不同？

**答题方向：**

1. 输出：传统确定可复现，AI 概率化不稳定
2. 延迟：AI 调用耗时秒级，需流式优化
3. 异常：隐性语义错误 + 显性代码错误双重问题
4. 成本：传统固定资源，AI 按 Token 按量计费
5. 观测：需要 Prompt 管理、LLM 链路追踪、质量评估

------

#### 🔴 如何控制 AI 应用的推理成本？

**答题方向：**

1. 长文本摘要压缩，精简上下文
2. 分级模型选型：复杂大模型、简单轻量模型
3. 语义缓存，复用重复问答结果
4. 合理控制 Chunk 数量、限制上下文长度
5. 用量监控 + 额度告警，防止超额消耗

------

#### 🟡 AI 应用如何做可观测性（Observability）？

**答题方向：**

1. LLM 调用链路追踪：耗时、Token、请求参数
2. Prompt 版本管理、灰度 A/B 测试
3. RAG 召回指标监控：相似度、召回数量
4. 模型输出质量评估、定期 Eval
5. 结合监控指标、日志、告警体系

------

#### 🟡 多轮对话如何管理上下文？

**答题方向：**

全量保留、滑动窗口、摘要压缩、向量记忆 四种方案按需组合使用，平衡上下文完整性与 Token 成本。

------

#### 🟢 如何对 AI 应用进行测试（Evaluation）？

**答题方向：**

1. 规则校验：格式、必填字段、规范约束
2. LLM-as-Judge：大模型交叉打分评审
3. 标准测试集回归，防止 Prompt 退化
4. 人工抽检 + 自动化评估工具结合

------
