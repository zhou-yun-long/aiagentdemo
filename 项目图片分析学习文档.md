## 2. 宣传图学习结论

### 2.1 产品定位

宣传图将产品定位为“AI 驱动用例管理平台”，核心价值是让测试用例从编写、组织、执行、评审到备份都可视化、自动化、可追溯。

图片中的主文案是“让用例管理更智能、更高效”，其下方说明强调“从用例编写到执行评审，全流程可视化管理，AI 助力质量提升”。这与项目需求文档中的 Treeify 定位高度一致：AI 驱动测试用例生成平台。

需要注意：宣传图品牌名为 `KCase`，项目需求文档品牌名为 `Treeify`。后续对外页面、Logo、标题栏、导出文件名需要统一命名。

### 2.2 核心功能链路

宣传图把平台能力组织成 5 个步骤：

1. 用例集管理：支持创建、组织、管理用例集，调整用例执行状态与流转进度。
2. 思路导图：用脑图方式组织测试场景，让用例编写更清晰。
3. AI 智能生成：基于自然语言描述或需求信息自动生成用例，支持一键导出或在线编辑。
4. 执行与评审：支持执行记录、结果标识、快速记录、流程闭环，保障用例质量。
5. 版本备份：自动备份用例版本，方便查看、对比与回滚，保证数据安全。

这说明产品不应只做“AI 生成文本”，而应覆盖“生成前输入、生成中结构化、生成后管理、执行后沉淀”的完整测试资产生命周期。

### 2.3 核心亮点

宣传图列出的亮点可以转化为前端首页或工作台的能力标签：

| 亮点 | 可转化功能 |
| --- | --- |
| AI 助力 | AI 生成、AI 续写、AI 解析、重新生成本阶段 |
| 可视化思路 | 思维导图、左侧大纲、节点折叠、标签视图 |
| 全流程管理 | 项目、用例集、用例、执行、评审、导出 |
| 多端适配 | Web 优先，后续考虑响应式与移动端只读查看 |
| 安全可追溯 | 版本历史、执行记录、操作日志、回滚 |

### 2.4 适用场景

宣传图明确覆盖功能测试、接口测试、性能测试、项目质量管理四类场景。前端标签体系中已有 `iOS`、`Android`、`Web`、`接口`、`数据`、`变更` 等标签，可继续扩展为测试类型与测试平台的筛选维度。

## 3. 前端截图学习结论

### 3.1 整体布局

截图中的界面是典型高密度工作台，不是营销式页面。主要结构为：

| 区域 | 作用 |
| --- | --- |
| 顶部状态栏 | 显示通过率、已测数量、进度条、标题、在线人数、分享、导出、全屏等全局状态 |
| 顶部工具栏 | 提供撤销、插入、移动、链接、图片、备注、工具、自动化、优先级、标签等编辑工具 |
| 左侧大纲 | 用树形列表辅助快速定位中心主题、分支主题与具体用例 |
| 中央画布 | 以思维导图呈现用例层级、前置条件、执行步骤、预期结果 |
| 右侧 AI 助手 | 接收自然语言输入，生成/续写/解析用例，并以表格结果辅助确认 |
| 底部工具 | 提供缩放、隐藏大纲、清除执行记录、快照当前结果等画布操作 |

当前前端代码已经基本采用该结构：`Toolbar`、`OutlinePanel`、`MindMapCanvas`、`AiAssistantPanel` 组成三栏工作台。

### 3.2 思维导图数据结构

截图中的用例树可抽象为如下层级：

```text
中心主题
  -> 功能/模块分支
    -> 测试用例
      -> 前置条件
      -> 执行步骤
      -> 预期结果
```

当前 `frontend/src/data/mindMap.ts` 中的 `MindNode` 已经覆盖核心字段：

| 字段 | 作用 |
| --- | --- |
| `id` / `parentId` | 表达节点树关系 |
| `kind` | 区分 root、group、case、condition、step、expected |
| `priority` | 标识 P0/P1/P2/P3 |
| `tags` | 标识前置条件、执行步骤、预期结果、AI、平台等 |
| `status` | 表达执行结果，当前包含 pass、warn、fail |
| `lane` / `depth` / `order` | 控制思维导图布局位置 |

后续如果接真实后端，建议保留这套前端视图模型，再增加接口层转换，不要让后端表结构直接绑死画布布局。

### 3.3 视觉风格

界面视觉以低饱和、强状态色、高信息密度为主：

| 元素 | 设计特征 |
| --- | --- |
| 背景 | 白色主题以浅灰/白为主，黑色主题以深灰为主 |
| 主色 | 蓝色用于选中、主要按钮、中心主题 |
| 状态色 | 绿色表示通过，红色表示失败，高优先级 P0 用红色强调 |
| 标签 | 多色轻量标签区分前置条件、执行步骤、预期结果、平台、AI |
| 节点 | 小卡片、细边框、轻阴影、圆角不大，适合密集阅读 |
| 线条 | 浅色曲线连接父子节点，黑色主题中使用偏金色线条增强可见性 |

当前 CSS 已经复用了这些特征：蓝色中心节点、P0/P1/P2/P3 标签、AI 标签、pass/warn/fail 状态、亮暗主题变量。

### 3.4 关键交互

截图中可以学习到的交互点：

| 交互 | 说明 |
| --- | --- |
| 主题切换 | 支持白色主题和黑色主题 |
| 节点编辑 | 通过插入下级、插入上级、移动、删除、链接、图片、备注等工具编辑用例树 |
| 标签编辑 | 顶部标签栏支持优先级、前置条件、执行步骤、预期结果、平台、AI 等快速标注 |
| 执行管理 | 顶部显示通过率和已测数量，节点显示通过/失败/警告等状态 |
| AI 生成 | 右侧输入需求描述，支持提交、引用 case、AI 续写、解析数据 |
| 结果确认 | AI 输出可以转成表格，用户再将结构化结果写回思维导图 |
| 导出分享 | 顶部提供导出用例、用例分享、在线人数等协作入口 |
| 画布控制 | 支持缩放、全屏、隐藏大纲、快照当前结果 |

## 5. 后续实现建议

### 5.1 产品叙事

对外宣传可以围绕“AI 生成 + 思维导图管理 + 执行评审 + 版本追溯”展开，避免只强调“聊天机器人”。这个项目真正的差异点是把 AI 生成结果落进可管理、可执行、可追溯的测试用例资产。

### 5.2 前端优先级

建议后续前端迭代按以下顺序推进：

1. 完成节点树 CRUD：新增、编辑、删除、移动、选中、折叠。
2. 完成 AI 助手到节点树的回填：自然语言输入 -> 结构化表格 -> 生成节点。
3. 完成执行状态：通过、失败、阻塞、未测、执行记录与通过率统计。
4. 完成导出：支持 JSON、CSV、Excel 或 Markdown。
5. 完成版本快照：保存当前结果、查看历史、差异对比、回滚。
6. 完成真实后端接入：项目、用例、生成任务、SSE、RAG 历史用例。

### 5.3 UI 设计原则

1. 保持工作台高密度，不做大面积宣传 Hero。
2. 顶部工具栏要为重复操作服务，按钮文案尽量短，图标优先。
3. 节点内容要稳定，不因选中、标签、状态变化导致画布明显跳动。
4. 标签和状态颜色要有明确含义，避免纯装饰色。
5. AI 助手应服务当前节点或当前用例集，不应成为独立聊天窗口。
6. 黑色主题需要单独校准线条、标签、表格边框的对比度。

## 6. 风险与待确认

| 问题 | 影响 | 建议 |
| --- | --- | --- |
| 品牌名不一致 | 宣传图是 KCase，项目文档是 Treeify | 先确定最终产品名，再统一前端标题、Logo、导出模板 |
| 图片带外部水印 | 不能直接作为正式产品资产使用 | 仅作为参考，正式宣传图应重新设计或使用自有截图 |
| 当前前端布局固定 | 数据量变化后可能重叠或超出画布 | 后续引入自动布局、拖拽布局或虚拟画布 |
| AI 输出未接后端 | AI 助手目前是静态演示 | 接入生成任务 API 与 SSE 后再完善交互 |
| 用例状态模型偏简单 | pass/warn/fail 不足以覆盖测试执行 | 增加未测、阻塞、跳过、执行中等状态 |
