# 📘 Treeify 项目面试题整理（高频 + 压测版）

---

## 🧩 一、项目整体介绍

### Q1：项目是做什么的？

**标准回答：**

Treeify 是一个 AI 驱动的测试用例生成平台，核心目标是将传统依赖人工经验的测试设计流程自动化。

系统基于多阶段 Agent 架构：
- E1：需求理解  

- E2：测试对象拆解  

- E3：测试场景生成  

  同时结合 RAG 检索历史用例和测试规范，提高生成结果与业务一致性。

  为了解决生成结果不稳定的问题，我设计了 Critic Agent，从覆盖率、路径完整性、可执行性等维度进行评估，并对低质量结果进行优化。

  在工程层面，通过 SSE 实现流式输出，让用户可以实时查看生成过程并进行逐步确认。



---

## 🧠 二、核心架构设计

### Q2：为什么要拆成三阶段（E1/E2/E3）？

**回答：**

单一 Prompt 无法稳定处理复杂任务，主要问题：

- 难以同时完成理解、拆解和生成  
- 中间结果不可验证  
- 可控性差  

所以拆分为：
- E1：语义理解  
- E2：结构建模  
- E3：推理生成  

本质是将复杂问题结构化。

---

### Q3：Sub-agent 和 多智能体有什么区别？

**核心结论：**

Sub-agent 是被编排的执行单元，多智能体是独立协作的个体。

**对比：**

| 维度     | Sub-agent | 多智能体    |
| -------- | --------- | ----------- |
| 控制方式 | 中心化    | 去中心化    |
| 执行方式 | pipeline  | 协作 / 对话 |
| 自主性   | 无        | 有          |
| 可控性   | 高        | 低          |

**回答模板：**

我项目中使用的是 Sub-agent，本质是一个可控的 pipeline。

多智能体更适合开放协作场景，而测试用例生成是强流程任务，所以选择 Sub-agent 更稳定。

## 🧠 三、Agent 深挖

### Q4：E2 测试对象拆解怎么做？

**回答：**

将需求转为结构化信息：

- 功能模块  
- 输入输出  
- 状态  
- 约束条件  

技术手段：
- JSON 结构化输出  
- Prompt 约束字段  
- Few-shot 示例  

---

### Q5：如果 E2 拆错了怎么办？

**回答：**

这是误差传播问题，我做了：

- 支持人工确认（可中断流程）  
- Critic 反向检查结构合理性  

后续可以优化：
- 多结果投票（self-consistency）

---

## 🔍 四、RAG 设计

### Q6：RAG 是怎么做的？

**回答：**

三层优化：

1. 数据层：文档分块  
2. 检索层：
   - 向量检索（pgvector）  
   - 关键词检索  
   - Top-K 融合  
3. Prompt 层：
   - 控制上下文长度  
   - 强约束使用检索内容  

---

### Q7：如何避免召回噪声？

**回答：**

- Top-K 控制数量  
- 相似度阈值过滤  
- 可加入 rerank  

核心是“精召回，而不是多召回”。

---

### Q8：RAG vs Fine-tune 为什么选 RAG？

**回答：**

- 数据更新频繁 → RAG 更灵活  
- 成本低  
- 可解释性更强  

Fine-tune 更适合稳定知识，不适合动态业务数据。

---

## ⚙️ 五、Critic Agent

### Q9：Critic Agent 是什么？

**回答：**

用于评估生成质量的 Agent，评估维度：

- 覆盖率  
- 路径完整性  
- 可执行性  
- 重复率  

低分结果会触发重试优化。

---

### Q10：如何避免“自己骗自己”？

**回答：**

- 不同 Prompt（角色分离）  
- 结构化评分  
- 阈值控制  

可优化：
- 多模型评审  
- 规则校验  

---

## 🌐 六、SSE 流式设计

### Q11：为什么用 SSE？

**回答：**

- 单向通信（服务端 → 前端）  
- 基于 HTTP，简单稳定  
- 自动重连  
- 适合 LLM 流式输出  

---

### Q12：断线怎么办？

**回答：**

- 前端自动重连  
- 记录 offset  
- 后端任务持久化（Redis/DB）  

实现断点恢复。

---

### Q13：高并发怎么办？

**回答：**

- 异步任务队列  
- 限流  
- 连接控制  
- Redis 缓存  

核心：削峰填谷

---

## 🗄️ 七、数据库设计

### Q14：为什么 MySQL + pgvector？

**回答：**

- MySQL：业务数据  
- PostgreSQL：向量检索  

通过 project_id 关联，实现：
- 业务查询 + 语义检索融合
