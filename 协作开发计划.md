# Treeify 协作开发计划

更新时间：2026-04-26

当前分支：`codex/backend-contracts`

## 1. 当前目标

先完成一个可演示、可继续扩展的 MVP：

输入需求 -> 后端生成任务 -> SSE 推送阶段结果 -> 前端脑图更新 -> 用例预览 -> 用例确认保存。

当前 mock 联调链路已经跑通，下一阶段重点是把 mock 能力逐步替换成真实能力，并把数据持久化。

## 2. 当前已完成

| 模块 | 状态 | 说明 |
| --- | --- | --- |
| 前端工作台 | 已完成 MVP | 支持需求输入、生成面板、脑图节点、用例预览 |
| 后端 mock API | 已完成 MVP | 项目、用例、生成任务、确认、SSE 均有 mock 接口 |
| SSE 分步生成 | 已完成 MVP | 支持自动模式和 E1/E2 分步确认 |
| 策略红包场景 | 已完成 mock | 输入红包/提现/tag 会生成对应脑图和用例 |
| Docker 部署 | 已完成本地版 | `docker compose up -d` 可启动前端和后端 |
| 仓库清理 | 已完成 | 已移除被 Git 跟踪的 `frontend/node_modules` 和 `frontend/dist` |

## 3. 建议人员分工

如果只有 1 个帮手，优先安排后端真实生成和数据库落库。

如果有 2 个帮手，可以这样分：

| 角色 | 负责人 | 主要目标 | 不建议改动 |
| --- | --- | --- | --- |
| 主线整合 | Codex | 接口契约、联调、SSE 主流程、代码合并 | 不单独长期占用业务模块 |
| 后端帮手 | 待定 | 真实生成服务、数据库表、Repository/Service | 不直接改前端 UI |
| 前端/部署帮手 | 待定 | 用例导出、脑图编辑体验、部署脚本、错误状态 | 不改后端核心编排 |

## 4. P0 开发任务

### 4.1 确认接口契约

目标：把 mock API 固化成真实后端也遵守的契约。

负责人：主线整合

参考文档：`speccase_前后端对接文档.md`

需要确认：

| 接口 | 说明 | 优先级 |
| --- | --- | --- |
| `GET /api/v1/projects` | 项目列表 | P0 |
| `GET /api/v1/projects/{projectId}/cases` | 用例列表 | P0 |
| `POST /api/v1/projects/{projectId}/generate` | 创建生成任务 | P0 |
| `GET /api/v1/generate/{taskId}/stream` | SSE 推送生成事件 | P0 |
| `POST /api/v1/generate/{taskId}/confirm` | 分步确认 | P0 |
| `POST /api/v1/cases/batch-confirm` | 确认保存候选用例 | P0 |

### 4.2 数据库持久化

目标：刷新页面后项目、需求、脑图、用例、生成任务不丢失。

负责人：后端帮手

建议先做这些表：

| 表名 | 说明 |
| --- | --- |
| `treeify_project` | 项目信息 |
| `treeify_requirement` | 用户输入的需求原文 |
| `treeify_generation_task` | 生成任务、阶段、状态、模式 |
| `treeify_generation_event` | SSE 事件流水，便于重放和排查 |
| `treeify_mindmap_node` | 脑图节点 |
| `treeify_test_case` | 测试用例 |

落库顺序：

1. 项目和用例先落库。
2. 生成任务和事件流水落库。
3. 脑图节点落库。
4. 再补版本号、软删除、审计字段。

### 4.3 真实生成服务

目标：保留 mock fallback，同时新增真实 AI 生成链路。

负责人：后端帮手

建议抽象：

```java
public interface TreeifyGenerationService {
    GenerateTaskDto createTask(Long projectId, CreateGenerateTaskRequest request);

    Flux<GenerateSseEventDto> streamTask(String taskId);

    GenerateTaskDto confirmStage(String taskId, ConfirmGenerateStageRequest request);
}
```

实现建议：

| 实现类 | 用途 |
| --- | --- |
| `MockTreeifyService` | 本地演示和 fallback |
| `AiTreeifyGenerationService` | 真实 AI 编排 |
| `PersistentTreeifyService` | 数据库存取 |

真实生成阶段：

| 阶段 | 目标 |
| --- | --- |
| E1 | 从需求中提取业务对象、页面、状态、规则 |
| E2 | 生成交互路径、异常路径、边界路径 |
| E3 | 生成测试用例标题、前置条件、步骤、预期结果 |
| Critic | 检查覆盖率、重复用例、缺失边界 |

### 4.4 前端工作台完善

目标：让用户能稳定完成一次工作，不被中断。

负责人：前端/部署帮手

优先任务：

| 任务 | 说明 | 优先级 |
| --- | --- | --- |
| 保存当前脑图 | 调用后端保存节点布局和内容 | P0 |
| 加载项目用例 | 进入页面自动加载项目和用例 | P0 |
| 用例确认后刷新 | 确认保存后同步更新脑图和表格 | P0 |
| 错误重试 | SSE 中断、后端异常时允许重试 | P0 |
| 导出用例 | 支持 CSV 或 Markdown 导出 | P1 |

### 4.5 部署验收

目标：任何人拉代码后可以快速启动。

负责人：前端/部署帮手

本地启动：

```bash
docker compose up -d --build
```

验收地址：

```text
http://localhost:5173/
```

基础验收：

1. 打开首页正常。
2. 输入策略红包需求。
3. 自动生成能跑完整流程。
4. 分步生成能停在 E1/E2 等待确认。
5. 用例预览内容和输入需求匹配。
6. 刷新页面后数据不丢失。

## 5. 每日协作规则

### 5.1 分支规则

| 类型 | 命名建议 |
| --- | --- |
| 主线联调 | `codex/backend-contracts` |
| 后端真实生成 | `feature/backend-ai-generation` |
| 数据库持久化 | `feature/persistence` |
| 前端体验 | `feature/frontend-workbench` |
| 部署 | `feature/docker-deploy` |

### 5.2 提交规则

提交信息建议：

```text
feat: add persistent generation task
fix: handle expected sse close in step mode
docs: update collaboration plan
chore: ignore frontend build outputs
```

每次提交尽量只包含一个模块，避免同时改前端 UI、后端服务和文档。

### 5.3 合并规则

合并前至少确认：

1. `npm run build` 通过。
2. 后端编译或测试通过。
3. `docker compose up -d --build` 能启动。
4. 浏览器能完成一次“输入需求 -> 生成 -> 预览”。
5. 没有把本地文档、环境变量、`node_modules`、`dist` 提交进去。

## 6. 帮手接任务时的第一件事

新加入的帮手先读这 3 个文件：

1. `协作开发计划.md`
2. `speccase_前后端对接文档.md`
3. `README.md`

然后只认领一个明确模块，例如：

```text
我负责后端数据库持久化，只修改 src/main/java 下 treeify 相关包、pom.xml、数据库 migration 文档。
```

不要多人同时改同一个文件，尤其是：

| 文件 | 原因 |
| --- | --- |
| `frontend/src/App.tsx` | 主工作台入口，冲突概率高 |
| `frontend/src/features/generation/useGenerateStream.ts` | SSE 主链路，冲突影响大 |
| `src/main/java/com/zoujuexian/aiagentdemo/service/treeify/MockTreeifyService.java` | 当前 mock 主服务，后续要拆分 |
| `docker-compose.yml` | 部署入口，容易互相覆盖 |
